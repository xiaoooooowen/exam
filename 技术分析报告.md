# 多端 TUI 早餐店系统 - 技术分析报告

## 一、项目概述

本项目要求实现一个多端 TUI（Terminal User Interface）系统，模拟早餐店从下单到制作到支付确认的完整业务流程。

### 1.1 核心需求

| 终端 | 端口 | 角色 | 功能特性 |
|------|------|------|----------|
| 叫号牌端 | 8024 | 只读 | 自动刷新当前叫号、制作状态、支付状态 |
| 顾客下单端 | 8025 | 可操作 | 多人连接、获取号码牌、下单、展示支付二维码 |
| 店主管理端 | 8026 | 可操作 | 查看订单列表、修改订单状态 |

### 1.2 关键约束

- 多端共享同一订单状态，状态变更需实时可见
- 订单状态流转必须合法（禁止非法跳转）
- 支持 SSH/Telnet 或其他终端协议

## 三、技术选型建议

### 3.1 终端协议方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **自建 TCP + 自定义协议** | 灵活、可控、无外部依赖 | 需要自行实现终端渲染 | 学习目的、定制化需求 |
| **SSH Server (libssh/Ganymed)** | 标准协议、安全、易用 | 依赖外部库、配置稍复杂 | 生产环境、安全要求高 |
| **Telnet** | 简单、实现容易 | 明文传输、无认证 | 内网/学习环境 |

**推荐方案**：自建 TCP + 自定义文本协议
- 降低实现复杂度
- 便于状态同步和实时推送
- 适合 TUI 场景的精细控制

### 3.2 技术栈建议

| 层级 | 推荐技术 | 理由 |
|------|----------|------|
| 后端核心 | Node.js / Go | 高并发、低延迟、事件驱动 |
| 网络通信 | Node.js net / Go net | 原生支持 TCP |
| 状态存储 | 内存 + JSON 文件 | 简单可靠、支持持久化 |
| TUI 渲染 | ANSI 转义序列 | 跨平台终端兼容 |

## 四、核心模块设计

### 4.1 订单状态机

```
┌──────────┐    支付成功    ┌──────────┐
│  待支付   │ ────────────► │  已支付  │
└──────────┘                └────┬─────┘
                                  │ 开始制作
                         ┌────────▼─────┐
                         │   制作中     │
                         └──────┬───────┘
                                │ 制作完成
                       ┌────────▼────────┐
                       │    已完成       │
                       └─────────────────┘
```

**状态定义**：
- `PENDING` - 待支付
- `PAID` - 已支付
- `COOKING` - 制作中
- `COMPLETED` - 已完成
- `CANCELLED` - 已取消

### 4.2 状态流转规则

| 当前状态 | 允许下一状态 | 触发条件 |
|----------|--------------|----------|
| PENDING | PAID, CANCELLED | 支付成功 / 用户取消 |
| PAID | COOKING | 店主开始制作 |
| COOKING | COMPLETED | 制作完成 |
| COMPLETED | - | 终态 |
| CANCELLED | - | 终态 |

### 4.3 核心数据模型

```typescript

  id: string;           // 订单号
```

## 五、状态同步方案

### 5.1 实时推送机制

```
                    ┌─────────────┐
                    │  事件总线   │
                    │ (EventBus)  │
                    └──────┬──────┘
                           │ 广播
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   ┌────▼────┐        ┌────▼────┐        ┌────▼────┐
   │ 叫号牌  │        │ 顾客端  │        │ 店主端  │
   │ 客户端  │        │ 客户端  │        │ 客户端  │
   └─────────┘        └─────────┘        └─────────┘
```

### 5.2 同步策略

| 场景 | 策略 | 实现方式 |
|------|------|----------|
| 订单状态变更 | 实时推送 | WebSocket / 长轮询 |
| 叫号更新 | 定时刷新 | 每秒广播当前状态 |
| 新订单创建 | 广播通知 | 事件驱动 |
| 断连重连 | 增量同步 | 记录最后状态版本 |

## 六、测试策略

### 6.1 必测场景

| 测试类型 | 测试内容 | 验证点 |
|----------|----------|--------|
| 多连接测试 | 多个客户端同时连接各终端 | 各终端功能正常、独立会话 |
| 状态流转测试 | 订单状态按合法路径变更 | 状态机正确、非法跳转被拒绝 |
| 断连恢复测试 | 客户端断开后重连 | 状态保持、数据不丢失 |
| 并发一致性测试 | 多客户端同时操作同一订单 | 状态一致、无竞态条件 |

### 6.2 测试用例设计

```python
# 测试用例示例
def test_状态流转():
    order = 创建订单()
    assert order.status == "PENDING"

    支付订单(order.id)
    assert order.status == "PAID"

    开始制作(order.id)
    assert order.status == "COOKING"

    完成制作(order.id)
    assert order.status == "COMPLETED"

def test_非法状态跳转():
    order = 创建订单()
    assert order.status == "PENDING"

    # 非法跳转：待支付 -> 完成
    result = 完成制作(order.id)
    assert result == False  # 应被拒绝
    assert order.status == "PENDING"
```

## 七、潜在风险与解决方案

| 风险 | 描述 | 解决方案 |
|------|------|----------|
| 并发冲突 | 多客户端同时修改订单状态 | 使用锁机制或乐观锁 |
| 状态不一致 | 网络延迟导致各端显示不一致 | 中心化状态管理+实时推送 |
| 断连丢单 | 客户端异常断开导致未保存 | 持久化存储+断连检测 |
| 内存泄漏 | 长连接未正确释放 | 连接池管理+超时机制 |
| 会话混淆 | 多用户会话数据混乱 | 每个连接独立会话ID |

## 九、总结

本项目是一个典型的**多端实时交互系统**，核心挑战在于：

1. **状态一致性** - 多端共享订单状态，需保证实时同步
2. **状态机合法性** - 严格控制订单状态流转，防止非法跳转
3. **并发控制** - 支持多人同时操作，需处理竞态条件
4. **TUI 交互** - 终端界面需做到高可读性和易用性

建议采用 **自建 TCP + 事件驱动** 的轻量级架构，实现简单且满足所有功能要求。
